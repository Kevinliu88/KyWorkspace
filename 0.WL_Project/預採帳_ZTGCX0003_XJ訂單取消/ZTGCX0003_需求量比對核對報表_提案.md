# XJ 需求量比對核對報表 — 提案（含「018 有、003 漏」漏料偵測 + 去重根因）

> **想法來源**（JosephLo, 2026-06-11）：XJ 上傳主要使用者是生管，他最大的痛是「**需求數量是系統給的，不知道對不對**」。做一支**唯讀比對報表**：對已建立的 XJ，**再呼叫一次 018（ZTGPPRP0019）**，把「建立當下存的需求量」與「即時重算的需求量」並排比差異，讓生管**自己警覺**到「建立當下數字極端異常」的單（如 1.3 vs 200），自行決定要不要整張取消重做。
>
> **本輪兩個收斂/新增**：
> 1. **只顯示差異、不做任何動作**（取消重做太複雜，先不碰）。
> 2. 順帶發現另一個問題：**018 跑得出來的料，003 不顯示**（新舊料：舊料不在現行 BOM，但仍有庫存要取消）——這份報表**也要顯示**，並**順帶定位根因在哪**。
>
> **結論先講**：兩件事其實是**同一支報表**——漏掉的舊料 = 比對的「018 有、003 沒有」那一桶，報表做出來就自動顯示，**並同時是漏料的偵測器**。報表**便宜、純唯讀、低風險，建議先做**；漏料的**根因在 `prepare_data` 的「最高版本料號去重」**，且該段在加總模式下**純損無益**，另案修。
>
> **姊妹文件**：`ZTGCX0003_XJ訂單取消_分析.md`（§12 B-7、§13 去重、§16 取數）、`預採帳_ZPRP018...\ZTGPPRP0019_計算回溯LOG_p參數提案.md`（log 機制，互補）。
> **地標慣例**：FORM 名 + 關鍵語句 + 表/欄名（不用行號）。

## 1. 背景與目的

- ZTGCX0003 的「需求數量」`bdmng` = 018（ZTGPPRP0019）展開的「需用料量」，**建立當下凍結、之後不刷新**（見分析文件 §16.4 B-7）。
- 實際踩過（A0510-000618）：建立時存 **1.831**，事後重跑 018 加總 = **1,441.679**。生管無從在當下察覺 1.831 是異常。
- **目的**：給生管一個**事後核對**的鏡子——不是要它判對錯（那要查 FM/業務拍板），而是讓「**建立當下極端異常**」的單**跳出來**：
  - 正常：`100 → 90 → 104`（小幅波動，不理會）。
  - 異常：`1.3 → 200 → 198`（建立時 1.3 明顯有問題）→ 生管警覺 → 人工取消重做。

> 驗證：A0510-000618（1.831 vs 1,441.679，比率 ~786×）這種單，這支報表**當場就會紅給你看**。

## 2. 核心洞察：兩個需求其實是同一支報表

把「建立時存的 003 集合」與「即時重跑的 018 集合」做 **full outer join**（鍵＝`idnrk + werks`），自然分三桶：

| 桶 | 意義 | 對應到使用者的需求 |
|---|---|---|
| **兩邊都有** | 003 存了、018 也有 | → **差異核對**（建立值 vs 即時值 + 比率 + 號誌） |
| **只在 018** | 018 跑得出、**003 沒存** | → **「漏掉的舊料」**（新舊料問題），🔴 標出 |
| 只在 003 | 003 有、018 現在沒有 | → 舊快照殘留 / 料已不在，標示供判讀 |

→ **一支報表同時滿足「比差異」與「補顯示漏料」**；而且「只在 018」那桶**就是漏料的偵測器**，哪張 XJ 漏了哪些料，一跑就現形。

## 3. 報表設計（純唯讀）

### 3.1 輸入
- `s_xj`（XJ 單號，可區間）為主；可選 `s_vtweg/s_werks/s_erdat` 縮範圍。
- **唯讀、零動作**：不寫 DB、不改狀態、不取消、不重做。

### 3.2 每張 XJ 的處理
1. 讀 `ZTCX0004`（該 XJ 各項次）→ **建立集合**：`{ (idnrk, werks) → bdmng（建立值）, stock_qty, canc_qty, vbeln, posnr }`。
2. 以該 XJ 的 `vbeln + posnr + canc_qty`，**再呼叫一次 018**：
   ```abap
   SUBMIT ztgpprp0019 WITH r_bt1   = 'X'
                      WITH p_vbeln = <vbeln>
                      WITH p_posnr = <posnr>
                      WITH p_menge = <canc_qty>   " 用建立時的取消量,保證同輸入
                      WITH r_mode1 = 'X'          " 加總(與 003 建立時一致)
                      WITH r_mode2 = space
     AND RETURN.
   ```
   + `cl_salv_bs_runtime_info` 攔截 → **即時集合**：`{ (matnr, werks) → menge（即時值） }`。
   > 完全複用 003 既有 `submit_pp029` 的寫法（同一支、同模式、同攔截）。
3. **Full outer join**（idnrk=matnr + werks）→ 組比對列。

### 3.3 輸出欄位（ALV）

| 欄位 | 來源 / 算法 |
|---|---|
| XJ 單號 / 項次 | ZTCX0004 |
| 料號 idnrk / 工廠 | join 鍵 |
| 建立時需求量 | ZTCX0004-`bdmng`（018-only 桶為空） |
| 即時 018 需求量 | 即時集合 menge（003-only 桶為空） |
| 差異 | `即時 − 建立`（絕對差） |
| **比率** | `即時 ÷ 建立`（建立=0 特例，見 §3.4） |
| **號誌燈** | 依比率門檻（§3.4） |
| 狀態 | `兩邊都有 / 003漏(舊料) / 018無` |
| 報庫存量 stock_qty | ZTCX0004（生管當初報的量，對照用） |
| XJ 狀態 xj_status / 建立日 | ZTCX0004 |

排序：**漏料(018-only) 置頂 → 再依比率偏離度降序**（最異常的在最上面）。

### 3.4 號誌門檻（用比率，不用絕對差）

> 為什麼用比率：`1.3→200` 是 153×、`100→104` 是 1.04×。絕對差會被大數量洗掉，**比率才抓得到「小數變大數」的極端異常**。

| 條件 | 號誌 | 意涵 |
|---|---|---|
| 003 漏掉（018-only） | 🔴 | 舊料未顯示，**漏報庫存** |
| 比率 ≥ 10× 或 ≤ 0.1× | 🔴 | 極端偏離，建議人工取消重做 |
| 1.5× ~ 10× / 0.1× ~ 0.67× | 🟡 | 注意 |
| 0.67× ~ 1.5× | 🟢 | 正常波動 |

- **建立值 ≈ 0**：除以小數會爆衝——**那正是訊號**（1.3 當分母 → 比率超大 → 紅）。`建立值 = 0 但即時 > 0` → 特例旗標「建立時 0、現有量」🔴。
- 門檻**先保守（10×）**，上線後依實際分佈調。

## 4. 可行性評估

| 面向 | 評估 |
|---|---|
| 重呼 018 | **複用** 003 既有 `submit_pp029`（SUBMIT + `cl_salv_bs_runtime_info`），零新機制 |
| 比對 | 兩內表 full outer join（SORT + READ / 雜湊），低 |
| 效能 | 一張 XJ = 一次 SUBMIT（一個 vbeln+posnr 重算所有子件）；幾秒/單，**適合 on-demand，勿做成每次顯示都自動跑** |
| 寫 DB | **無**，純記憶體比對，零資料風險 |
| 對 003 主流程 | **不影響**（獨立報表）或僅加唯讀按鈕 |

→ **技術可行性高、風險低。**

## 5. 本質限制（必須誠實）

| 限制 | 說明 |
|---|---|
| **它是啟發式警示，不是對錯判定** | 大落差 = 「值得人看一眼」，不是「一定錯」。最終靠人判斷（已是你的設計意圖）。 |
| **只抓「會自己好」的暫態異常** | 若建立當下的怪值是**確定性 bug**（018/FM 對該料每次都算 1.3），重跑也是 1.3 → 落差 0 → **false negative**。它抓 timing/transient（如 MRP 半跑、計畫單未生），抓不到 deterministic bug。 |
| **分不清 glitch vs 合理大幅漂移** | 訂單量大改、BOM 改版也會觸發 → 是警示不是攔截。 |
| **基準本身不完美** | 即時 018 加總可能 over-count（分攤列 + 9999/8888 直接預留列同料相加，見 B-7）→ 但對抓「**極端**異常」仍足夠（1.3 vs 1441 還是爆表）。🟢 不代表「數字一定對」。 |
| **時機是關鍵** | 落差來自「建立↔核對」之間 MRP 變動。**上傳當下立刻按 = 同一 MRP 狀態 = 落差≈0 = 看不出**。需**隔一段時間 / 下次 MRP 跑完再核對**才有偵測力 → UI 要顯示「建立日 vs 核對時間」讓人有感。 |

## 6. ★ 漏料根因分析：`prepare_data` 最高版本去重

**「018 有、003 沒有」的料，不是 018 漏給、也不是 save_data 砍的，而是 `prepare_data` 那段字串去重砍的。**

### 6.1 砍在哪

`prepare_data` 的 `LOOP gt_pp029 INTO ls_029` 開頭，有一段以料號**結尾版本字母**算「同基礎碼最高版本 `lv_best`」的邏輯，結尾：

```abap
" 只處理最高版本
IF ls_029-matnr NE lv_best.
  CONTINUE.            " ← 不是「最高版本」就整列跳過 = 丟掉,不進 gt_data
ENDIF.
```

→ 把「**同基礎碼、只差結尾版本字母**」的料（如 `A0510-000618` ↔ `A0510-000618A`）當成重複，**只留一個、CONTINUE 掉另一個**。新舊料只要差一個版本字母，就被砍掉一個；**舊料雖不在現行 BOM、卻有實體庫存要取消**，就這樣被靜默丟掉。

### 6.2 為什麼這段在 003「純損無益」

- 003 呼叫 018 **固定用加總模式**（`submit_pp029` 帶 `r_mode1='X'`）。
- 018 加總 = `SUM(menge) GROUP BY (kdpos, del12, werks, matnr)`，而 `del12` 是常數（FM 內 `ls_mldelay_index_1-rcenr`）→ 回給 003 的 **每個 `(matnr, werks)` 本來就只有一列**。
- 所以 003 這段去重**收不到任何「真重複列」可合併，唯一作用就是跨砍不同料號的版本變體** → **沒有任何 dedup 效益，只有砍掉合法舊料的傷害**。

### 6.3 邏輯本身也脆弱

該段 `無版本字母` 的 ELSE 分支會反過來偏好沒字母的基礎碼，使「砍新還砍舊」**順序相依、不穩定**——但**效果一致：新舊版本對一定砍掉一個**。

### 6.4 從「待確認」升級為「確認的問題」

此即分析文件 **§13** 標「最高版本去重未限 werks、待業務確認（別預設立場）」那條。**本輪業務案例給了答案**：舊料有庫存要報 → **不該砍** → 這條從「待業務確認」升級為**確認需修**。

## 7. 兩件事分開處理（報表 vs 修根因）

| 事 | 內容 | 風險 | 順序 |
|---|---|---|---|
| **A. 比對報表（本提案主體）** | 純唯讀顯示差異 + 漏料三桶 | 低（不寫 DB、不碰主流程） | **先做** |
| **B. 修去重砍舊料** | `prepare_data` 那段在加總模式無益只有害 → **移除 / 大幅收窄**（只在真有重複 matnr 才合併，不跨砍版本變體） | 中（需先確認「這段原本想解什麼」，避免誤拆；建議用 A 報表先把實際被砍的料攤出來當證據） | 後做、另案 |

> **A 是 B 的證據來源**：報表先把「到底砍了哪些料、影響多大」蒐集出來，B 才改得有根據、不誤拆。
